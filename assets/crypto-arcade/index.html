The Three Fundamental Properties of XOR
1. Associativity — $(a \oplus b) \oplus c = a \oplus (b \oplus c)$
Grouping doesn't matter — lets us rearrange terms
2. Self-Inverse (Involution) — $a \oplus a = 0$
Anything XOR'd with itself cancels to zero
3. Identity (Zero Element) — $a \oplus 0 = a$
XOR with zero changes nothing
These three properties appear in every symmetric cipher: AES (AddRoundKey), ChaCha20 (stream XOR), HMAC, hash functions, and the one-time pad.

The Four Primitives
OWF — Easy forward, hard to invert. The foundation.
PRG — Stretch short seed into long pseudorandom stream.
PRF — Keyed function: input → pseudorandom output.
PRP — A reversible PRF: every input maps to a unique output, and you can always decrypt back. AES is a PRP.

Block by Block
A block cipher encrypts fixed-size blocks. Identical input blocks produce identical output. Click blocks to encrypt them.
$E_k: \{0,1\}^n \rightarrow \{0,1\}^n$ — AES uses $n = 128$ bits (16 bytes)
What is a block cipher?
Unlike stream ciphers (Room 2) that encrypt one byte at a time, a block cipher chops your message into fixed-size chunks (blocks) and encrypts each one. AES always uses 128-bit blocks (16 bytes) regardless of key size. Think of it like a machine that can only process exactly 16 letters at a time — no more, no less.
$$E_k(\text{block}_i) = E_k(\text{block}_j) \iff \text{block}_i = \text{block}_j$$
Identical blocks → identical ciphertext. This is the fundamental weakness we'll exploit next. What if your data has patterns?
The Penguin Test
The most famous demo in crypto. Toggle between ECB and CBC mode to see why ECB is catastrophic for structured data.
What is ECB mode?
ECB (Electronic Codebook) is the simplest mode: encrypt each block independently with the same key. The problem? If two plaintext blocks are identical, their ciphertexts are also identical. For images, this means spatial patterns survive encryption — you can still see the shape even after encrypting.
ORIGINAL
ENCRYPTED (ECB)
$$\text{ECB: } C_i = E_k(P_i) \quad \text{(independent)}$$
ECB encrypts each block independently. Identical plaintext blocks produce identical ciphertext blocks, preserving spatial patterns. This is why you can see the penguin!
Chain the Blocks
CBC mode chains each block to the previous one. Step through to see how identical blocks get different ciphertext.
$C_i = E_k(P_i \oplus C_{i-1})$ — each block depends on all previous blocks
How CBC fixes ECB
CBC (Cipher Block Chaining) XORs each plaintext block with the previous ciphertext block before encrypting. This means even if two plaintext blocks are identical, they get XORed with different values — producing different ciphertext. The first block uses a random IV (Initialization Vector) since there's no "previous ciphertext" yet. Step through below to see each stage.
$$C_0 = E_k(P_0 \oplus IV), \quad C_i = E_k(P_i \oplus C_{i-1})$$
The IV randomizes the first block. The chain propagates. Even identical plaintext blocks produce different ciphertext because they XOR with different previous blocks.
The IND-CPA Game
You pick two messages. The oracle encrypts one. Can you tell which? Against ECB: always. Against CBC: never.
What is IND-CPA?
IND-CPA (Indistinguishability under Chosen Plaintext Attack) is the standard security definition for encryption. The game: you give an oracle two messages $m_0$ and $m_1$. It flips a coin, encrypts one, and returns the ciphertext. If you can guess which message was encrypted with probability better than 50%, the scheme is broken. Deterministic encryption (like ECB) always fails this — encrypt $m_0$ yourself and compare!
$$\text{Deterministic} \implies \text{NOT IND-CPA}$$
You beat ECB by submitting the same message twice and comparing ciphertexts -- they're identical because ECB is deterministic. CBC defeats this because a fresh random IV means the same plaintext encrypts differently every time. This is why every secure mode must use randomization.

Flip the Bits
Encryption alone doesn't prevent modification. Click ciphertext bytes to flip bits — watch the plaintext change predictably.
Stream cipher: $c = m \oplus k$. Flip any bit in $c$ → the same bit flips in $m$.
What does "malleable" mean?
An encryption scheme is malleable when an attacker can modify the ciphertext in a way that causes a predictable change in the plaintext — without knowing the key. With stream ciphers (XOR-based), flipping any bit in the ciphertext flips the exact same bit position in the decrypted plaintext. The attacker has surgical control over what changes.
CIPHERTEXT (click bytes to corrupt)
DECRYPTED PLAINTEXT
$$\underbrace{c'}_{\text{modified ciphertext}} = c \oplus \underbrace{\Delta}_{\text{your flip}} \implies \text{Dec}(c') = m \oplus \Delta$$
$c'$ = the ciphertext after you flipped bits. $\Delta$ = the exact bits you changed. Because XOR is its own inverse, your flip passes straight through decryption into the plaintext. The attacker controls the change without knowing the key.
The Bank Heist
An encrypted bank transfer says "$100". You need it to say "$900". Flip the right bit to change the amount.
The attack math
You want to change '1' (0x31) to '9' (0x39). Compute the difference: $\Delta = 0\text{x}31 \oplus 0\text{x}39 = 0\text{x}08$. Now XOR the ciphertext byte at that position with $\Delta$. When decrypted: $m \oplus k \oplus \Delta \oplus k = m \oplus \Delta$ — the key cancels, your change applies directly to the plaintext. Find the byte for '1' and flip it.
INTERCEPTED TRANSFER
CIPHERTEXT BYTES (click to flip the right one)
$$c \oplus \Delta \;\;\text{decrypts to}\;\; m \oplus \Delta$$
The key cancels out completely. You chose $\Delta = 0\text{x}08$ because $\text{0x31} \oplus \text{0x39} = \text{0x08}$. The attacker controls the plaintext change without ever knowing the key.
Add the Tag
Same bank transfer, but now with a MAC tag. Try the same attack — see what happens.
$\text{Tag} = \text{MAC}_k(m)$ — keyed integrity check
What is a MAC?
A MAC (Message Authentication Code) is a keyed hash. The sender computes $t = \text{MAC}_k(m)$ using a secret key $k$ and sends $(m, t)$. The receiver recomputes the MAC with the same key and checks if it matches. If the message was tampered with, the tag won't match — the receiver rejects. The attacker can't forge a valid tag without the key. Security notion: EUF-CMA (existentially unforgeable under chosen message attack).
PROTECTED TRANSFER
CIPHERTEXT (try to tamper)
$$\text{Verify: } \text{MAC}_k(m') \stackrel{?}{=} t \implies \textbf{REJECT}$$
The MAC catches every modification. If even 1 bit changes, the tag won't match. The receiver rejects the message. Integrity is preserved.
The Order Matters
Three ways to combine encryption and MAC. Only one is fully secure. Click each to see why.
The three compositions
We need both encryption (confidentiality) and MAC (integrity). But the order you combine them matters hugely:
• Encrypt-and-MAC: Encrypt $m$, MAC $m$ separately. Problem: MAC might leak info about $m$ (it's computed on plaintext).
• MAC-then-Encrypt: MAC $m$, then encrypt $m \| t$. Problem: you must decrypt before you can check the tag — processing potentially tampered data.
• Encrypt-then-MAC: Encrypt $m$, then MAC the ciphertext. Tampering caught immediately — never even decrypt bad data.
$$\text{Enc-then-MAC:} \quad c = E_k(m), \quad t = \text{MAC}_{k'}(c)$$
MAC covers the ciphertext, so tampering is caught BEFORE decryption. The other orders either leak information (Enc-and-MAC) or can't detect tampering early (MAC-then-Enc).

The Problem
Alice and Bob need a shared secret key. But Eve is watching everything they send. How do you agree on a secret over a public channel?
The key distribution problem
Everything we built in Rooms 1–4 (XOR, stream ciphers, block modes, MAC) requires Alice and Bob to share a secret key. But how do they agree on that key in the first place? If they're across the internet, they can't meet in person. Any key sent over the network is visible to eavesdroppers. This chicken-and-egg problem held back cryptography until 1976, when Diffie and Hellman found a brilliant solution.

The "paint mixing" is modular exponentiation: $g^a \bmod p$. Easy to compute forward (fast exponentiation), but given $g^a \bmod p$, finding $a$ requires solving the Discrete Logarithm Problem (DLP) — believed computationally infeasible for large primes. Eve Attacks
DH is secure against passive eavesdroppers, but active attackers can perform a Man-in-the-Middle attack.
What is a MITM attack?
A Man-in-the-Middle (MITM) attack exploits the fact that plain DH has no authentication. Eve intercepts Alice's $g^a$ and replaces it with her own $g^z$. She does the same with Bob's $g^b$. Now Eve has a shared key with Alice ($g^{az}$) and a separate shared key with Bob ($g^{bz}$). She can decrypt Alice's messages, read them, re-encrypt for Bob, and neither party knows. Step through below to see exactly how.
Eve has TWO separate shared keys — one with Alice, one with Bob. She can decrypt, read, re-encrypt, and forward. Neither Alice nor Bob knows. DH alone is NOT enough!

The key insight
Non-repudiation = the sender cannot deny having sent the message. MACs fail here because the shared key means either party could have created the tag. We need something only Alice can produce, but anyone can verify. That's a digital signature.
$$\text{MAC (shared key)} \implies \text{no non-repudiation}$$
Digital signatures fix this: Alice signs with her private key (only she has it). Anyone verifies with her public key. If verification passes, only Alice's private key could have produced that signature.

Forge This!
You're the attacker now. You have Alice's public key and one valid signed message. Can you forge a signature on a different message?
Your mission
You've intercepted a valid signature. Now you want to create a fake signature on a different message that will pass verification — without knowing the private key.

The problem: you need to find $\sigma$ such that $\sigma^7 \bmod 77 = \text{target}$. That means computing $m^{1/7} \bmod 77$ — which requires knowing $d$!
Hint
Without the private key $d$, your only option is brute force — try every $\sigma$ from 0 to 76 and check if $\sigma^e \bmod n$ equals the target.

With our toy numbers ($n = 77$), that's only 77 guesses. With real RSA (2048-bit keys), $n$ has over 600 digits — brute force would take longer than the age of the universe.

Chain of Trust
Signatures prove who signed something — but how do you know a public key actually belongs to who it claims? Welcome to PKI.
The trust problem
You visit bank.com. The server sends a public key and says "I'm your bank." But an attacker could do the same thing! You need a trusted third party to vouch for the server's identity.

A Certificate Authority (CA) signs a digital certificate that says: "This public key really belongs to bank.com." Your browser comes pre-loaded with trusted Root CAs. Click each certificate below to walk the chain.
$$\text{Root CA} \xrightarrow{\text{signs}} \text{Intermediate CA} \xrightarrow{\text{signs}} \text{Server Cert}$$
Trust is transitive. You trust Root CAs (pre-installed in your OS/browser). They vouch for Intermediates. Intermediates vouch for servers. Break any link and the chain fails — your browser shows a certificate warning.
